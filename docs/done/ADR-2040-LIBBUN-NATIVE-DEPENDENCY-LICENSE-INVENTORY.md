# ADR-2040: libbun Native Dependency License Inventory

Status: Done
Date: 2026-05-18

The native plugin links Bun's native runtime stack, including JavaScriptCore and
WebKit-derived components. Bun's top-level license note identifies Bun itself as
MIT-licensed while separately calling out JavaScriptCore/WebKit LGPL
requirements.

`libbun` cannot treat that as a one-time README fact. Each native plugin release
needs an auditable inventory of the exact native dependency set shipped in that
release, because the plugin binary is the artifact downstream distributors may
choose to redistribute.

## Decision

`libbun` release automation must generate and publish a native dependency
license inventory for every native plugin binary release.

The inventory must be derived from release inputs rather than handwritten from
memory. At minimum it must record:

- the `BUN_SOURCE_COMMIT` value;
- the Bun license note used for the release;
- Bun's native link manifest entries consumed by the plugin build;
- WebKit and JavaScriptCore source origins, versions or commits, and license
  notices;
- any LGPL-covered, GPL-covered, Apache-2.0, MIT, BSD, ICU, OpenSSL-family, or
  other third-party native libraries linked into the plugin;
- whether each dependency is source-only, statically linked into the plugin,
  dynamically linked as a separate system library, or redistributed as a
  separate binary;
- where the complete corresponding source for each redistributed dependency is
  provided.

The inventory must be checked into or attached to the same release bundle as the
plugin compliance package from ADR-2039. For GitHub publication, the inventory
must be attached to the same GitHub Release as the plugin binary and source
archive.

The inventory must be generated by the official GitHub Actions release workflow.
Handwritten inventories may be used as reviewed source inputs, but the published
release inventory must be produced from the same checked-out tag, Bun source
pin, native link manifest, and plugin binary that produced the release asset.

## Version Policy

Where a dependency says "LGPL 2 or later" or equivalent, the release notes must
state which version `libbun` is using for compliance analysis.

Where a dependency is LGPL 2.0-only, unknown, or ambiguous, `libbun` must use a
conservative release posture until the ambiguity is resolved. For the native
plugin, that means publishing enough source and build material for users to
modify the LGPL-covered dependency and rebuild a compatible replacement plugin.

`libbun` must not describe downstream obligations as "dynamic linking makes it
fine" without also identifying the exact LGPL version and the replacement/source
path supplied by the release.

## Required Tooling

The release process includes `scripts/package-native-plugin-release.sh`, which:

- reads `BUN_SOURCE_COMMIT`;
- reads Bun's generated native link manifest;
- records the license inputs for `libbun`, Bun, JavaScriptCore, and WebKit;
- records the native libraries named by Bun's link manifest;
- emits the exact Git commit, release tag, plugin ABI version, Bun source
  revision, and plugin binary checksum;
- emits a machine-readable inventory file and a human-readable notice file;
- fails release verification if required source, link-manifest, or binary inputs
  are missing.

The GitHub Actions release workflow must run that script before uploading
assets, include its generated files in the release checksums, and fail the
release if the generated inventory differs from any committed expected
inventory template.

The generated output should be stable enough to diff across releases, so
dependency changes are visible during review.

The inventory must include enough information for a downstream distributor to
copy the inventory into its own third-party notices without reverse-engineering
the `libbun` build. It should also include a short "Redistribution Notes"
section that names the required companion artifacts from the same GitHub
Release.

## Consequences

The inventory turns LGPL compliance from tribal knowledge into release data.

Downstream distributors can point to the `libbun` release's plugin inventory and
source bundle instead of independently reconstructing which Bun/WebKit/JSC
artifacts were linked.

The intended downstream handoff is:

```text
This product optionally uses libbun's native plugin.
Plugin binary: <exact libbun GitHub Release asset URL>
Corresponding source: <same release source archive URL>
Notices and licenses: <same release notice/inventory URLs>
Checksums: <same release checksum URL>
Build workflow: <same release/source link to the GitHub Actions workflow>
Replacement: configure the product to load a compatible replacement plugin.
```

`libbun` maintainers still need to review the inventory when Bun is updated. A
new Bun source commit can change native dependencies, license texts, build
scripts, or the WebKit/JSC source correspondence required for a complete plugin
compliance bundle.

## Implementation

This ADR is implemented by `scripts/package-native-plugin-release.sh` and the
official `.github/workflows/release-native-plugin.yml` workflow.

The packaging script records the release version, repository commit,
`BUN_SOURCE_COMMIT`, plugin ABI version, platform, plugin binary checksum,
Bun/JSC/WebKit license inputs, the native link manifest entries, and
redistribution notes in `licenses.json`. It also emits a human-readable
`NOTICE.txt`, `SOURCE.txt`, and release checksum file.

The workflow runs the packaging script before release upload and attaches the
generated inventory to the same GitHub Release as the native plugin binary and
source archive. If a committed expected inventory template is present, the
script compares the generated inventory against it so dependency changes become
reviewable release diffs.
