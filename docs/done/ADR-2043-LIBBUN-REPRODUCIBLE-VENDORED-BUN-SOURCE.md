# ADR-2043: Reproducible Vendored Bun Source

Status: Done
Date: 2026-05-18

## Context

`libbun` vendors a pinned Bun source snapshot under `vendor/bun` and applies a
small libbun-specific patch set so Bun can be embedded through the native
plugin. Some upstream Bun paths are ignored by Bun's own `.gitignore`, and
some dependency source is added by libbun scripts after the upstream archive is
extracted.

That creates a maintenance risk: a local `git add vendor/bun` can silently omit
ignored upstream source files, and an edit directly inside `vendor/bun` can
become indistinguishable from upstream source. For LGPL-oriented source
delivery and future Bun upgrades, the vendored source must be regenerated by
scripts rather than hand-edited.

## Decision

The committed `vendor/bun` source snapshot must be reproducible from:

- `BUN_SOURCE_COMMIT`;
- `scripts/apply-vendored-bun-patches.sh`;
- patch files under `patches/vendored-bun/`;
- `scripts/vendor-bun-deps.sh` for explicitly scripted extra dependency
  source.

No libbun change may be made directly in `vendor/bun` unless it is also
represented by a patch file or deterministic script that can be replayed after
re-vendoring.

Ignored upstream source files must still be tracked when they are part of the
pinned upstream Bun tree. Ignored scripted dependency source must also be
tracked when that dependency is part of the committed source snapshot.

Build outputs, caches, downloaded build-time dependency checkouts, generated
bindings, and local package-manager output remain untracked.

## Implementation

`scripts/apply-vendored-bun-patches.sh` now accepts `LIBBUN_VENDOR_DIR` so the
same patch application path can be used against the real vendored tree and a
temporary verification tree.

`patches/vendored-bun/0001-add-libbun-native-plugin-pic-build-mode.patch`
contains the libbun PIC-mode changes to Bun's build scripts.

`scripts/vendor-bun-deps.sh` now accepts `LIBBUN_VENDOR_DIR` and remains the
single source of truth for explicitly scripted dependency source currently
committed under `vendor/bun/vendor`.

`scripts/stage-vendored-bun-source.sh` stages only reproducible source files:
the pinned upstream Bun file list and explicitly scripted dependency source. It
uses `git add -f` for those paths so upstream-ignored source files are not
silently omitted, while local build products remain untracked.

`scripts/verify-vendored-bun-reproducible.sh` rebuilds a temporary source tree
from the pinned Bun commit, applies libbun's patch/dependency scripts, checks
out the committed vendored source snapshot, overlays any current working-tree
vendored changes, and fails if the two source trees differ.

## Consequences

Vendored Bun upgrades now have a stricter workflow:

```sh
scripts/update-vendored-bun.sh <ref-or-sha>
scripts/stage-vendored-bun-source.sh
scripts/verify-vendored-bun-reproducible.sh
```

This may force-track files that Bun itself ignores, including test fixture
`node_modules` directories and dotfiles, when those files are present in the
pinned upstream source commit. That is intentional: the committed source
snapshot must not depend on ignore-rule accidents.

If a future dependency is added under `vendor/bun/vendor`, the dependency must
be generated by `scripts/vendor-bun-deps.sh` or another deterministic script
called from the re-vendor path before it is committed.

## Acceptance Criteria

- libbun-owned Bun modifications are represented by patch files or scripts;
- upstream-ignored Bun source files from `BUN_SOURCE_COMMIT` are force-tracked;
- explicitly scripted dependency source is force-tracked;
- local build outputs remain untracked;
- `scripts/verify-vendored-bun-reproducible.sh` passes.
